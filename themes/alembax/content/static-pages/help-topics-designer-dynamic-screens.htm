Dynamic Screens

Dynamic screens allow fields and sections of the screen that an analyst or user is completing to be made available depending on dynamically changing conditions.  You can configure rules that will make parts of the screen hidden, read-only or optional until certain conditions are met, at which point they become visible, optional or mandatory.

This helps to keep the screen simple, and reflect the order in which you want things done, helping the analyst or user complete the task in hand quicker.

You can apply these to most of the elements that make up a screen – fields (including HTML fields), headings, and entire tabs or sections. Buttons can even be hidden until required, according to your needs. Wherever the properties Hidden, Read Only or Required appear on a screen element, this new feature can be used.

Examples of Dynamic Screens in Action

Here are just some ways in which you can use dynamic screens. Click on the links to find out more about each scenario:

  • display a warning if a call is about to breach
  • hide non-essential fields when users are logging a Critical incident
  • only show implementation details on a Change after it is Authorized
  • hide unnecessary detail on a Service form until you need to add it to the catalog

Priority Override

It is common to not only want to audit changes to call Priority, but also to restrict who can change it in the first place. Dynamic fields can be used to control this, as described below. (This assumes that the Priority Matrix is in use, but can be adapted if it is not.)

  • Make Impact and Urgency (which drive Priority) available when logging calls, but only available when updating calls if an analyst checks Override Priority
  • Make Override Reason visible and mandatory if Override Priority is checked

To implement this, two rules are needed:

  • New Call
{:Open} = “New Call”
  • Not New and Not Overridden
 {:Open} <> “New Call” AND {:Override Priority} = “False”

Apply the rules as follows:

  • Set Read Only on Impact and Urgency to Not New and Not Overridden
  • Set Read Only on Override Priority to New Call
  • Set Hidden on Override Reason to Not New and Not Overriddenden

Additionally:

  • Set Required on Override Reason to Always
  • Set Access on Override Priority to Read Only for the Security Profiles who should not be able to change Priority

Simpler Screens

You can simplify the layout of screens by making fields hidden until activated by a checkbox. A simple example is to hide the Caused by RFC number (and any associated fields from the Change) on an Incident, unless the Caused by RFC checkbox is selected. This is achieved by applying a rule to the Hidden property on the dependent fields, which is applied if '{: Caused by RF} = "False" '.

You can ensure that change approval is received for knowledge changes and thatreviews are performed, by using two checkboxes - Change Approved and QA Complete. The setting of these is audited, and they become visible and mandatory when moving the article status from Draft or Review Required to . This is enabled using the following four rules:

  1. Draft - {:Entry Status} = "Draft"
  2. Not Draft - {:Entry Status} <> "Draft"
  3. Not Published - {:Entry Status} = "Draft" OR {:Entry Status} = "Review Pending"
  4. Published - {:Entry Status} = "Published Internally" OR {:Entry Status} = "Published to Portal"

The settings on these fields are as follows:

Change Approved

Hidden if Draft Required if Not Draft

QA Complete

Hidden if Not Published Required if Published

The same technique can be used to audit the confirmation of other actions, such as notifying Service Stakeholders that a knowledge entry related to a Service has been published.

Setting up Dynamic Screens

Dynamic screens are easy to set up.

  1. Define the rule in ASM Designer.
  2. Apply it to as many fields as you want.

If you define the rule on a parent screen, it will be available to all its child screens, should you wish to use it.
