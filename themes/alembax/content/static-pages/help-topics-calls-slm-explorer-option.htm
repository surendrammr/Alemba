SLM Explorer Option

The SLM explorer option allows you to view Service Level Management agreements currently in effect on a call. If an agreement is in effect against a call an orange indicator appears before the SLM option on the Call Details Explorer.

Additional SLM actions can be performed through the Service window, such as suspending or resuming an agreement (also referred to as stopping or starting the clock on the agreement) or closing events on all linked agreements when the service has resumed.

Viewing linked agreements

  1. Display the Call Details window, if it is not already on screen.
  2. In the Explorer pane, expand Details, then select SLM.
  3. The Call Service Level Management window appears showing any agreements that are currently in effect against the call. The SLA, OLA, and UC fields display which agreements have been applied to the call currently. There can be up to three agreements simultaneously in effect, though only one of each type. The lower part of the window contains a table listing all agreement events for this call. There are three different types of agreements that get linked to calls:

  • SLAs: agreements with users
  • OLAs: agreements with internal groups
  • UCs: agreements with external suppliers
  • Select the agreement hyperlink to view the Agreement Details window.

    • ASM automatically applies agreements to calls based on a combination of information (the call details, IPK selection criteria, agreement selection priority, and the action performed by the analyst).

    Applying SLAs when a call is logged

    When a call is logged and then saved, deferred or forwarded, ASM tries to find a matching agreement (SLA) where field values specified in call details match IPK Selection Criteria on any SLAs. When multiple SLAs are matched, ASM will use the agreement with the matching IPK Selection Criterion that has the higher priority (defined in the Agreement Selection Priority in SLM administration). If no matches are found, ASM checks for a default SLA and applies it, otherwise no SLA is applied. 

    Applying OLAs

    When a call is logged and then forwarded internally, ASM tries to find a matching agreement (OLA) where the group selected on the Forward Internally window matches the group selected in the IPK Selection Criteria on any OLAs. If a match is found, the OLA is applied to the call, otherwise, no OLA is applied.

    Applying UCs

    The rules for forwarding externally and selecting UCs are more complex. UCs are only invoked if you forward a call externally to an analyst who is linked to the contract matching the UC Selection Criteria, that is, they are the point of contact for the external supplier. If the UC is linked to both an External Supplier and Contract through the Selection Criteria, then both the External Supplier and Contract selected on the Forward Externally window must match those linked to the UC. Otherwise, if the UC is only linked to either an External Supplier, or Contract, then, only one of these needs to match when forwarding the call externally.

    Applying SLAs when a call is updated

    If no SLA was initially linked to the call when it was logged, ASM tries to find a matching agreement when the updated call is saved, deferred, or forwarded in the same way as if you applied the SLA when a call is logged.

    If an agreement is now applied, the target times for all events will be based on the original time the call was logged, not the time you update the call.

    If an SLA was initially applied to the call, the new SLA invoked is determined by the Lock IPK SLA setting in the Agreement Details.

    • If the option is selected, the original agreement applied to the call will always be assigned regardless of any changes made to the fields of the call, even if those details were used to select the agreement in the first place.

      • If after logging a call with a certain user, you decide to change the user for that call assuming you made a mistake when the call was logged, ASM will not change the agreement on the call. The original agreement for the first user will still be in effect. An analyst with sufficient permissions will have to change the agreement manually on the Call Service Level Management window

    • If the option is cleared, and the original agreement assigned by the system still applies (that is, the original agreement has not been manually changed), the Agreement Selection Criteria is reassessed each time the call is updated and deferred/forwarded/saved. If a new matching agreement is found, the current agreement is closed and the new agreement is applied. If the same agreement still applies, then the existing agreement continues to be applied to the call. If no matches are found, ASM checks for a default agreement and applies it, otherwise no SLA is applied and the original agreement is closed.

    If no SLA was initially applied to a call, selecting the Lock IPK SLA setting in the SLA agreement details does not prevent the SLA from being applied to a call if the call is updated.

    When a new agreement is applied to a call, keep in mind that the target times for all events will be based on the original time the call was logged, not the time you changed the agreement over.

    You log a call and Agreement A is applied to the call automatically. According to its matrix, it will breach the agreement in one hour. Half an hour later, you change over to agreement B, which also has a time of 45 minutes until breach. However, changing the agreement has not “reset the clock”; you now only have 15 minutes before the agreement is breached, even though it has just come into effect on the agreement.

    CMDB items and agreements

    If needed, when multiple CMDB items (config items, services, service actions, and service bundles) are linked to a call, ASM will evaluate the CMDB Selection Priority list defined in administration. This enables the system to determine which CMDB item to use for finding and applying an agreement.

    If required, you can override an agreement that the system has applied to the call through manually changing the agreement.

    You must have Service Agreement Override selected in the Options tab of your IPK Management security role.

    1. If you know the name of the agreement, type the entire name or part of it in the SLA, OLA, or UC field depending on the type of agreement you want to attach. Alternatively, use , use the button to search for and find the agreement.
    2. Once an SLA is selected, the Information Panel on the Call Details window displays the name of the selected SLA.

    This table shows all the events being calculated for the agreements on this call. The columns displayed can be configured using the Columns toolbar button, but can include the following:

    • Agreement Ref – the reference code for the agreement.
    • Title – the title of the agreement.
    • Type – the type of the agreement.
    • Activity – a code describing the actual event.
    • Target Time – the time the event will take place (or has taken place).
    • Status – the current status of the event, such as Active (open), Closed (completed), Suspended (when the clock on the agreement is temporarily stopped), Resumed (when the event resumes after a suspension), and Failed.
    • Closing Event – if the event has been closed, this describes what caused that to happen (closed, call updated, agreement manually changed, or clock restarted).
    • Description – if the clock was stopped, this field shows the reason entered as to why this happened.
    • Start Time – this is the time when the agreement starts recording events against the call.
    • Exclude – this column appears if SLM Exclusions have been configured by your system administrator and enables you to exclude selected events from SLM reporting.

    Events are color-coded to indicate whether they are Active/Closed (appear in black), Suspended (appear in blue), and Failed (appear in red).

    Events that were closed when a stopped clock is started again will have the Status “Resumed” and Closing Event “Clock Started”. This identifies that the events have been resumed by new events.

    To view the old events in the Events table, select Include Closed Events.

    Excluding Events from SLM Reports

    You must have SLM Exclusions selected in the Options tab of your IPK Management security role.

    If you have the appropriate security role permissions, an additional column, Exclude, appears in the browse table. This enables you to hide selected SLM events from appearing in SLM reports. To exclude selected SLM events, follow these steps:

    1. Select the checkbox in the Exclude column.
    2. Click Apply.
    3. If exclusion reasons are configured in the SLM Settings, choose a reason for excluding the event.
